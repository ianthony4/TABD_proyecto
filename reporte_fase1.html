Reporte de Fase 1: Scaffolding y MVP A

Entorno de pruebas automatizado para el estudio comparativo de recuperación de información médica.

Contexto del Proyecto

Se ha construido la infraestructura base (scaffolding) orientada a objetos para evaluar y comparar dos enfoques arquitectónicos frente a bases de conocimiento médico (Documentos del MINSA):

  • MVP A: Recuperación tradicional (RAG Vectorial con Embeddings y ChromaDB).
  • MVP B: Recuperación Vectorless / Tree Reasoning (Estructuración mediante PageIndex).

El diseño garantiza una integración limpia, mediante el uso de interfaces abstractas, permitiendo intercambiar el modelo de recuperación (Retriever) sin modificar la lógica del orquestador.

Estructura y Archivos del Proyecto

src/interfaces/retriever_interface.py

Contiene la clase abstracta BaseRetriever. Define un contrato estricto con los métodos index(pdf_path) y query(question). Esto es vital para el polimorfismo del orquestador.

src/mvp_a_vectorial/chunker.py

Módulo de procesamiento de PDFs usando pdfplumber. Se encarga de dividir el texto extraído en fragmentos superpuestos para no perder contexto entre las divisiones.

src/mvp_a_vectorial/embedder.py

Gestiona la creación de embeddings con el modelo local all-MiniLM-L6-v2 de Sentence-Transformers y la persistencia efímera en la base de datos vectorial ChromaDB.

src/mvp_a_vectorial/vector_controller.py

Implementa BaseRetriever. Orquesta exclusivamente el pipeline del MVP A, inyectando el Chunker y el Embedder para ofrecer una interfaz unificada de indexación y búsqueda.

src/orchestrator.py

El bucle central del Test Harness. Recibe un retriever activo, lee las preguntas ground-truth desde un archivo CSV con pandas, ejecuta la consulta y almacena los resultados esperados vs obtenidos.

src/llm_client.py

Capa de abstracción (Mock) para el LLM. Actualmente simula la generación de respuestas. En el futuro, alojará las peticiones HTTP al API de Google MedGemma.

src/evaluator.py

Contiene la clase estática encargada de calcular métricas. Se ha implementado el scaffold para la métrica base de validación cruzada (Exact Match).

main.py

Punto de entrada de la aplicación. Crea los datos simulados de forma dinámica e inicializa el pipeline inyectando la dependencia de MVP A en el Orquestador.

✅ Logros de la Fase 1 (MVP A)

Completado
  • Arquitectura Sólida: Aplicación estricta de Principios SOLID, permitiendo la escalabilidad para inyectar nuevos MVPs en el futuro con cero impacto al código existente.
  • Pipeline RAG Funcional: Se implementó la lógica completa desde la extracción del PDF, normalización (chunking con overlap), creación de embeddings vectoriales y búsqueda por distancia (Cosine Similarity).
  • Automatización del Test Harness: El orquestador es completamente funcional; es capaz de iterar dinámicamente archivos CSV, invocar búsquedas de contexto y generar un reporte de resultados procesable automáticamente por pandas.
  • Mocking Preparado: La transición a un LLM real requiere de la modificación de un solo método, gracias al diseño del cliente MockMedGemmaClient.
